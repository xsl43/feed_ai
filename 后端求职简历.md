#### 项目概述

从零构建的类抖音/微博短视频信息流平台后端，支撑前端 Feed 流、用户社交互动、AI 视频分析三大核心场景。采用 Go + Gin 单体分层架构，集成 MySQL / Redis / RabbitMQ / MinIO 全套基础设施，支持 Docker Compose 一键部署，API 层与异步 Worker 拆分部署。

**技术栈**：Go 1.24 · Gin · GORM · MySQL 8.0 · Redis 7 · RabbitMQ 3 · MinIO · Docker Compose · JWT · SSE

#### 功能模块详述

##### 1. 账号与鉴权体系
- **JWT 双模式鉴权**：硬鉴权（需登录接口）与软鉴权（Feed 等公开接口，有 token 则标注身份，无 token 则匿名访问），无感切换
- **Token 防重放**：签发时将 token 同步写入 Redis + MySQL，每次验证双写比对，登出即撤销
- **Refresh Token 机制**：access token 15min 过期后，前端自动用 refresh token 换取新 token，用户无感知
- **限流保护**：登录接口按 IP 限流 10次/min，注册按 IP 限流 5次/h，防暴力破解

##### 2. Feed 流核心引擎（最具技术深度）

**冷热分离 + 游标分页**：
- **热数据**：新发布视频写入 Redis ZSET `feed:global_timeline`，按创建时间排序，毫秒级查询
- **冷数据**：热度下沉的视频降级到 MySQL，通过 `singleflight` 合并请求防缓存穿透
- **冷热缝合**：当热数据不足时自动拼接 MySQL 冷数据补全，游标 `latest_time` 跨冷热边界无缝翻页
- **ZSET 灾备重建**：Redis 清空时用 `singleflight` 全局锁从 MySQL 重建 1000 条时间线

**热度排行**：
- 滑动窗口聚合 60 个分钟级快照 (`ZUnionStore`)，实现滚动 1 小时热度榜
- 快照 TTL 2 分钟，热点翻页期间内快照稳定不抖动

**缓存体系**：
```
L1 本地缓存 (go-cache, 3-5s TTL)
  → L2 Redis (JSON序列化, 1h TTL)
    → L3 MySQL (兜底查询 + singleflight 防击穿)
```
- `GetVideoByIDs` 实现批量三级缓存查询，L2 命中则异步回写 L1
- `ListByFollowing` 使用 Redis 分布式锁防缓存击穿，锁失败则轮询等待

##### 3. 社交互动模块

**点赞 / 评论 / 关注** — 异步落库 + 最终一致性：

```
用户操作 → API 立即响应 → 同步写 DB(+ 乐观锁) 
         → 投递 RabbitMQ → Worker 异步消费 → 更新计数 + 热度
```

- **Outbox 发件箱模式**：视频发布时先写 MySQL `outbox_msgs` 表，`OutboxPoller` 每秒轮询投递到 RabbitMQ，`TimelineConsumer` 消费后写入 ZSET 并保留最近 1000 条
- **死信队列（DLX）**：消费失败自动路由至 DLX，超过 `MaxRetryCount` 后丢弃并记录日志
- **热度积分**：点赞 +1、评论 +2（带权重的 popularity 字段），用于热度排行
- **幂等性**：`IgnoreDuplicate` 确保重复消息不会重复计数

##### 4. AI 智能分析模块

- **运行时配置**：API Key / Model / BaseURL 可在运行时通过 API 动态更新，无需重启，`sync.RWMutex` 保证并发安全
- **视频全链路分析**：FFmpeg 提取音频 → SiliconFlow ASR 语音转文字 → DeepSeek 智能总结（结构化 Markdown 输出）
- **异步分析**：提交后立即返回，Worker 用指数退避重试（3 次，2s / 4s / 8s）处理外部 API 的不稳定性
- **令牌桶限流**：全局 AI 调用限 10次/min，防止 API Key 额度耗尽

##### 5. 通知系统（SSE 实时推送）

- 自定义 SSE Hub：`map[userID][]chan` 管理连接，支持同一用户多端在线
- 30s 心跳保活，`http.Flusher` 实时推送
- 点赞/评论/关注事件通过 RabbitMQ Topic 路由 → `NotificationWorker` 消费 → `SSEHub.Push` 投递

##### 6. 媒体文件管理

- **MinIO 分片上传**：支持断点续传，每片 5MB，`upload_id` 追踪会话
- **MD5 秒传**：上传前检查 MD5，已存在则秒传不重复占用存储
- **FFmpeg 集成**：视频转音频、转码，超时 15min 自动 Kill

##### 7. 容器化部署

- Docker Compose 编排 7 个服务：MySQL、Redis、RabbitMQ、MinIO、API Server、Worker、Nginx 前端
- 多阶段 Dockerfile（build → base → api/worker），最终镜像基于 Alpine，无编译环境
- 健康检查依赖链：API 等待 MySQL/Redis/RabbitMQ/MinIO 全部 healthy 后才启动

## 关键技术决策

| 问题 | 方案 | 理由 |
|------|------|------|
| Feed 流翻页一致性 | ZSET 游标分页 + 冷热边界缝合 | 避免 MySQL OFFSET 深分页性能问题 |
| 缓存穿透 | `singleflight` 合并并发请求 | 防止 Redis 过期瞬间大量请求击穿到 MySQL |
| 缓存击穿 | 分布式锁 + 轮询等待 | 避免热点 Key 过期时重复计算 |
| JWT 密钥一致性 | `sync.Once` 缓存密钥 | 修复了每次调用生成新密钥导致 token 签发即失效的 bug |
| 视频热度排行 | Redis ZUnionStore 滑动窗口 | 无需维护大表 counter，天然支持滚动窗口 |
| API 防抖 | 前端 Axios 拦截器 + `isRefreshing` 锁 | 防止多个 401 并发导致 token 刷新竞争 |
